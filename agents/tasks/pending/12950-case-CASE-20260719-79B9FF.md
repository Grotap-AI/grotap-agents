## Requirements
LT2-C: latency + error-rate observability so degradation is visible WITHOUT a load test.

CONTEXT (2026-07-19 golive-1003 load test): API p50 degraded from ~1.5s to ~5s under load and 659 5xx accrued, and nothing alerted — the only reason we know is a JSONL from the test harness. /health stayed green (~500ms) the whole time, so the current probe cannot see request-path degradation.

REQUIRED WORK:
1. Timing middleware (or extend the APISEC request_log if it already records duration): capture per-request duration_ms + status. Keep overhead trivial (no extra DB writes on the hot path — aggregate in-process via shared_state counters/histograms, flush periodically).
2. Rolling stats endpoint GET /dashboard/system-metrics (grotap-admin gated): last-15-min p50/p95 latency, request count, 5xx count, 429 count, active strike blocks. Feed the existing /machine or system-health UI panel if one fits.
3. Alerting: extend the 15-min pe-watch (agent-06 cron) to also alert when p95 > threshold (env, default 3000ms) or 5xx rate > 1% over the window. Reuse the existing email path + label. Alert must state the top slow endpoints.
4. Strike/auto-block events surface in the same 15-min sweep (count + orgs affected) — complements LT2-B's immediate email for paying orgs.
5. Tests: middleware records; endpoint math; alert threshold trip; py_compile + boot test.

ACCEPTANCE: rerunning even a 10-user smoke visibly moves the metrics endpoint, and an artificial slow endpoint (test-only) trips the alert path in a unit test. RELEASE: master -> prod.

## Clarifications
- Q: What is the target p95 latency threshold default (you mention 3000ms for alerting—is that the production SLA)?
  A: 3000ms is the ALERT threshold default, not a production SLA — it sits deliberately between normal p95 at low load (a few hundred ms) and the post-improvement load-test target (p95 < 4000ms at 50 concurrent, p50 < 1500ms, from LT2-E acceptance). Make it env-tunable (PE_WATCH_P95_ALERT_MS, default 3000) so it can be tightened as the platform speeds up. There is no formal SLA document; treat the LT2-E targets as the current capacity bar.
